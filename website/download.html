Installation Guide

Get
Started

Set up BrowserShield in 6 steps. The backend, ML model, and Chrome extension all need to be running together.

Prerequisites
Python 3.8+
Google Chrome
pip installed
Git (optional)
1
📁
Get the Project Files

Download or clone the BrowserShield project to your computer.

Terminal / CMD
$ git clone https://github.com/yourusername/browsershield.git
$ cd browsershield
2
🐍
Create Virtual Environment

Isolate the project dependencies from your system Python installation.

Terminal / CMD
$ python -m venv venv
# Windows
$ venv\Scripts\activate
# Mac / Linux
$ source venv/bin/activate
3
📦
Install Dependencies

Install all required Python packages in one command.

Terminal / CMD
$ pip install -r requirements.txt
Successfully installed flask scikit-learn pandas numpy ✓
4
🤖
Train the ML Model

Run the training script once to generate the phishing detection model. Creates email_model.pkl and vectorizer.pkl.

Terminal / CMD
$ cd ml_training
$ python train_email_model.py
✓ Model saved: models/email_model.pkl
✓ Evaluation report: models/evaluation_report.txt
$ cd ..
5
⚗️
Start the Backend Server

Run the Flask backend. Keep this terminal open — the extension communicates with it while you browse.

Terminal / CMD
$ cd backend
$ python app.py
Starting BrowserShield on http://0.0.0.0:5000 ✓
💡 Dashboard available at http://127.0.0.1:5000/dashboard
6
🧩
Install the Chrome Extension

Load the extension in Chrome using Developer Mode — no Chrome Web Store required.

1
🌐
Go to chrome://extensions/
2
🔧
Enable Developer Mode toggle (top right)
3
📂
Click Load unpacked → select extension/ folder
4
🛡
BrowserShield icon appears in Chrome toolbar ✓
Verification

Check Everything Works

Four things to confirm before you start browsing.

🟢
Backend Running
Visit http://127.0.0.1:5000/ — JSON status response appears.
🟢
Dashboard Loads
Visit /dashboard — forensics incident table appears.
🟢
Extension Connected
Click the toolbar icon — popup shows Backend Connected in green.
🟢
Detection Working
Browse any website — badge updates and incidents appear in the dashboard.
Troubleshooting

Common Issues & Fixes

These are the real problems encountered during development and testing of BrowserShield — with the exact root cause and fix applied in each case.

❌
ISSUE 01
Extension Cannot Reach Backend
Symptom Popup shows "Backend Offline". Console: net::ERR_CONNECTION_REFUSED
The Flask server is not running or the terminal running it was closed. The extension requires the backend to be active at all times.
✓ Fix
Navigate to the backend/ folder and run python app.py. Keep the terminal open — the server must stay running while browsing.
🤖
ISSUE 02
ML Model File Not Found on Startup
Symptom Backend crashes: FileNotFoundError — models/email_model.pkl not found
The model has not been trained yet. The models/ folder exists but .pkl files are missing — they are generated by the training script, not included in the repo.
✓ Fix
Run: cd ml_training then python train_email_model.py. This generates both email_model.pkl and vectorizer.pkl in the models/ folder.
🗄️
ISSUE 03
Database Saves in Wrong Location
Symptom Dashboard shows no incidents even though backend logs confirm they are saved. Two incidents.db files exist in different folders.
The original db.py used a relative path "incidents.db". The database was created wherever Flask was run from — not in the database/ folder. This created two separate databases.
✓ Fix Applied in Code
Changed to absolute path using os.path.abspath(__file__) so the database always saves at backend/database/incidents.db regardless of where the app is launched from.
🌐
ISSUE 04
CORS Error — Extension Requests Blocked
Symptom Console: Access blocked by CORS policy. No 'Access-Control-Allow-Origin' header present on response.
The browser extension runs on a different origin than the Flask server. Without CORS headers, the browser blocks all cross-origin requests silently.
✓ Fix Applied in Code
Added CORS headers to all responses in app.py using @app.after_request hook. Access-Control-Allow-Origin: * now allows the extension to communicate freely with the backend.
📡
ISSUE 05
VMware Extension Cannot Reach Host Backend
Symptom Extension in virtual machine fails to connect. Backend works on host but not from VMware browser.
Extension was connecting to 127.0.0.1 (localhost). Inside VMware, localhost points to the VM itself — not the host machine running Flask.
✓ Fix
Change BACKEND in background.js from 127.0.0.1 to the host machine IP (e.g. 192.168.x.x:5000). Flask already listens on 0.0.0.0 so it accepts external connections.
🔁
ISSUE 06
Same URL Logged 3–5 Times Per Visit
Symptom Dashboard shows duplicate incidents for the same URL. Backend receives multiple requests per single page navigation.
Chrome fires webNavigation.onCompleted multiple times per page load — once for the main frame and again for sub-frames and redirects. Each event triggered a separate API call.
✓ Fix Applied in Code
Added a URL deduplication cache (JavaScript Set) in background.js. Each URL is cached after first analysis. Duplicate events within the session are skipped. Cache is capped at 200 entries.
🔬
ISSUE 07 — Critical
Sandbox Always Returned "Safe" — Hardcoded Result
Symptom Every downloaded file — including invoice.pdf.exe and malware.exe — was reported as Low Risk / Safe regardless of content
Root Cause 1: sandbox.py was completely empty — the file existed but contained zero code. In file_routes.py, the sandbox result was hardcoded as sandbox_result = "safe" meaning every file passed as safe unconditionally.
✓ Fix Applied
Built sandbox.py from scratch with 7-step static analysis: MD5 hash lookup, magic byte detection, Shannon entropy scoring, 35 suspicious code pattern scans, PE header analysis, extension spoofing detection, and file size anomaly checks.
Root Cause 2: Even after writing sandbox.py, the route was still not calling it. The hardcoded result remained in file_routes.py — the sandbox function was never actually invoked.
✓ Fix Applied
Rewrote file_routes.py to call run_sandbox(file_path) when risk score ≥ 4. The sandbox verdict can now upgrade risk level — files passing extension analysis can still be flagged High Risk if malicious patterns are found inside.
📧
ISSUE 08
Phishing Emails Passing Sender Check
Symptom Phishing emails from evil@attacker.com that mentioned "PayPal" in the body were passing the sender domain check as legitimate.
The original code checked if the sender domain appeared anywhere in the body text. Mentioning a brand name in the email content would pass the check — completely wrong logic.
✓ Fix Applied in Code
Changed to extract actual link domains from URLs in the body using regex, then compare the sender domain against those link domains — not the raw text. Only a real URL domain match now passes the check.
📄
ISSUE 09
Double Extension Spoofing Not Detected
Symptom File named invoice.pdf.exe was treated as low risk. The dangerous .exe extension was hidden behind a decoy .pdf extension.
Original file_analysis.py only used os.path.splitext() which returns only the final extension — making double extension attacks completely invisible to the system.
✓ Fix Applied in Code
Added _check_double_extension() function that splits the filename on all dots, detects when a high-risk extension follows a low-risk decoy, and immediately assigns a score of 5 — flagging it as High Risk.
